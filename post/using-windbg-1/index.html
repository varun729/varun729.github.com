Using WinDbg-1

Behind all the UI of Windows, there still exists software that is running. And when you are writing software it is bound to break and you want to debug it. GDB on Linux has helped me debug programs. Although, it took me some time to get used to understanding GDB, I am able to understand some of the intrinsics of how it works and can use it for debugging programs. I want to be able to debug Windows programs with the same ability, especially the programs that I write.

I am starting to use WinDbg to debug executables on Windows. I am using my Windows machine to learn and program in Go. While programming, I want to start learning a debugging tool as well, and I find that WinDbg is a good option. This series of articles would be an \“experience\” article, where I will jot down how I am using WinDbg and the tricks that I am learning. I hope that in a few months, I can do some of the basic debugging tasks on WinDbg with ease even when debugging Go programs.

View

WinDbg is GUI tool that can load an executable for debugging.

  1. File > Open Executable…
    • loads the executable in memory
    • opens a Command window where we interact with the debugger (step, go, etc.)
  2. View > Disassembly
    • opens a Disassembly window where the instructions around the current instruction are listed
    • if we enter a function, the code in the Disassembly windows updates to the code in the function

Commands

There are several debugger commands to interact with the debugger. I am only listing a few here that I am starting to use.

  1. Go (g) : This starts executing the program. This command looks similar to the ‘continue’ or ‘run’ command in gdb.
  2. Step Into (t) : This command allows to step into the program. Thus, we execute one instruction at a time. Very useful to analyze problematic code.
  3. Call stack (k) : This command prints the call stack. It is useful to find the control flow.

Info

When we load the executable, command window displays the following information:

CommandLine: \“####exe_name####\”

Symbol search path is: srv*

Executable search path is:

ModLoad: 00000000 00400000 00000000 00677000 image0000000000400000

Here image00000000 is the name of the memory region of the program executable.
